# 钉钉群聊自然语言数据查询机器人设计文档

## 1. 项目目标
在钉钉群聊中通过 `@机器人 + 自然语言` 完成数据库查询，并支持两类输出：

1. **即时问答流**：直接在群里返回文本摘要和简单表格。
2. **分析看板流**：当问题属于长时间跨度、同比/环比、趋势、原因归因、退款/投诉分析等场景时，生成结构化 JSON，并推送到网站固定 HTML 模板看板，同时给出 AI 分析结论与建议。

## 2. 已确认约束
- 使用 **同一个钉钉机器人**。
- 机器人运行在**用户本机**。
- 钉钉接入方式使用 **Stream Mode**，不依赖公网回调地址。
- 已具备机器人凭据和数据库连接信息。
- 使用 **阿里百炼 / 千问模型**。
- 查询范围为 **单数据库全库**。
- 第一版支持 **汇总统计 + 明细检索 + 分析类问题**。
- 群内 **所有人都可使用**。
- 数据库账号**无法改成只读**，因此必须在机器人服务端实现强风控。
- 结果中敏感字段按 **部分脱敏** 返回。
- 回复中 **不显示实际 SQL**。
- 当问题语义不清时：**能确定就直接答，不确定再追问**。
- 回复形式：**文本摘要 + 简单表格**；复杂分析同步触发 **HTML 看板**。

## 3. 典型问题示例
- 上周销售数据怎么样？
- 上周同比去年销售怎么样？
- 最近有多少新用户？
- 最近投诉的人多不多？
- 用户投诉主要都因为哪些原因？
- 最近退款因为什么原因？

这些问题说明系统不仅要支持检索和聚合，还要支持：
- 时间窗口解析（最近、上周、本月等）
- 同比/环比
- TopN 分析
- 原因归因
- 趋势总结与建议生成

## 4. 整体方案
采用 **单机器人 + 单后端 + 双工作流共享内核**：

### 4.1 工作流 A：即时问答流
处理短问题、直接查询、简单统计、明细检索。

流程：
1. 钉钉群内 `@机器人` 发问。
2. Stream 客户端收到消息。
3. 系统抽取问题、用户、会话上下文。
4. AI 解析意图并输出查询任务草案。
5. 服务端根据数据库元数据和安全规则生成/校验 SQL。
6. 执行查询。
7. 结果脱敏、整理、总结。
8. 在群内回复文本摘要 + 简单表格。

### 4.2 工作流 B：分析看板流
处理 7 天、30 天、同比、环比、原因分析、趋势分析、退款/投诉归因等问题。

流程：
1. AI/规则识别为分析型问题。
2. 服务端拆解为一个或多个统计查询。
3. 生成看板所需的结构化 JSON。
4. 调用 AI 对结构化结果做分析和建议生成。
5. 把 JSON 保存/推送到网站固定模板所需位置。
6. 群里返回摘要说明 + 看板链接。

## 5. 模块设计

### 5.1 钉钉接入模块
职责：
- 连接钉钉 Stream Mode
- 接收群消息事件
- 判断是否 `@机器人`
- 发送文本回复

输入：钉钉事件
输出：统一消息对象（群、用户、文本、时间、会话）

### 5.2 消息编排模块
职责：
- 规范化用户问题
- 管理会话上下文
- 路由到问答流或看板流
- 在必要时发起追问

路由判断依据：
- 是否包含长时间跨度（7天/30天/本月/上周等）
- 是否涉及同比/环比/趋势/原因/建议
- 是否需要复杂统计或多阶段分析

### 5.3 元数据管理模块
职责：
- 启动时扫描数据库元数据（表、字段、类型）
- 可定时刷新元数据缓存
- 结合 `表映射.xls` 作为中文别名与业务提示源

说明：
- `表映射.xls` 不是唯一数据源，只作为辅助增强。
- 系统应具备全库自发现能力，以支持“全库可查”。

### 5.4 AI 语义理解模块
职责：
- 识别问题意图（统计/明细/趋势/原因/同比）
- 识别时间范围、指标、维度、过滤条件
- 输出结构化查询计划，而不是直接信任模型执行 SQL

结构化输出示例字段：
- question_type
- metric
- dimensions
- filters
- time_range
- comparison_type
- candidate_tables
- candidate_fields
- requires_dashboard
- clarification_needed

### 5.5 SQL 生成与安全控制模块
职责：
- 根据结构化计划和元数据构造候选 SQL
- 对 SQL 做严格安全校验
- 拦截高风险语句和越权访问

必须执行的限制：
- 只允许单条 `SELECT`
- 禁止 `INSERT` / `UPDATE` / `DELETE` / `REPLACE`
- 禁止 `DROP` / `ALTER` / `TRUNCATE` / `CREATE`
- 禁止多语句执行
- 禁止注释绕过
- 禁止访问高风险系统库
- 默认加 `LIMIT`
- 限制最大返回行数、列数、执行时间

由于数据库账号不是只读账号，**这一层是系统安全核心**。

### 5.6 查询执行模块
职责：
- 建立数据库连接池
- 执行安全校验通过的 SQL
- 对超时、空结果、异常做统一处理

### 5.7 结果处理模块
职责：
- 结果转表格
- 结果摘要生成
- 敏感字段部分脱敏
- 统一回复格式化

默认脱敏示例：
- 手机号：138****1234
- 姓名：按规则保留部分字符
- 身份证号/银行卡号：中段打码

### 5.8 看板数据生成模块
职责：
- 将分析型问题转为固定 JSON 数据结构
- 为网站固定 HTML 模板提供可直接渲染的数据文件
- 生成看板标题、时间范围、核心指标、趋势、TopN、原因分布等字段

建议 JSON 结构包括：
- title
- generated_at
- time_range
- summary_metrics
- trend_series
- comparison_metrics
- top_items
- reason_breakdown
- ai_summary
- ai_suggestions

### 5.9 审计与日志模块
职责：
- 记录用户问题、识别出的意图、执行耗时、结果规模、是否触发看板
- 不记录明文敏感结果
- 对失败原因做可追溯记录

## 6. 交互规则

### 6.1 直接回答
适用场景：
- 指标清楚
- 时间范围清楚
- 目标表/字段可推断

例如：
- 最近有多少新用户？
- 上周销售数据怎么样？

### 6.2 追问确认
适用场景：
- “最近”无法从上下文安全推断
- “销售”可能指销售额、订单数、支付人数等多个候选指标
- 条件明显缺失且无法默认

遵循规则：
- 能确定就直接答
- 不确定再追问

## 7. 网站看板设计边界
- 看板由**固定 HTML 模板**承载。
- 机器人后端不动态生成页面结构，只生成数据和分析内容。
- 支持把 JSON 文件落地到本地目录、再同步到网站，或通过网站接口上传。
- 群内只展示摘要和看板链接，不在群里输出大段分析正文。

## 8. 技术选型建议
推荐第一版使用 Python：
- 钉钉 Stream SDK / 等价接入库
- 数据库客户端（按实际数据库类型选择）
- LLM 调用封装（百炼/千问）
- 轻量 Web 服务（如 FastAPI）用于：
  - 健康检查
  - 本地预览接口
  - 看板 JSON 输出或对接网站 API

选择 Python 的原因：
- 本地部署简单
- 生态成熟
- LLM/数据库/钉钉接入都比较方便

## 10. 关键风险与应对

### 风险 1：账号非只读
应对：
- 强制 SQL 白名单能力（仅 SELECT）
- 语法解析和规则双重校验
- 禁止多语句
- 独立执行账户封装

### 风险 2：模型幻觉导致查错表/字段
应对：
- 基于真实元数据约束生成
- 模型先产出结构化计划，再由程序映射
- 查不到时宁可追问，不盲查

### 风险 3：全库可查导致口径混乱
应对：
- 元数据缓存 + 中文别名映射
- 给高频指标预置语义规则
- 用 few-shot 示例规范时间和指标解析

### 风险 4：分析问题链路长
应对：
- 拆成“查询计划 → SQL → 聚合结果 → AI总结 → 看板JSON”多阶段
- 将每阶段输出持久化，便于排障

### 风险 5：本机部署稳定性一般
应对：
- 用 Stream Mode 避免公网回调依赖
- 增加自动重连、日志、启动脚本
- 后续可平滑迁移到服务器

## 11. 第一版范围（MVP）
第一版必须包含：
- 钉钉 Stream Mode 接入
- 百炼/千问调用
- 单数据库元数据自动扫描
- 即时问答流
- 分析看板流
- 固定模板看板 JSON 输出
- 敏感字段部分脱敏
- SQL 安全控制
- 审计日志

第一版暂不追求：
- 多数据库
- 复杂权限分级
- 图表模板自动生成器
- 动态前端页面生成
- 多轮会话深记忆

## 12. 实施顺序
1. 创建项目骨架与配置加载
2. 打通钉钉 Stream 接入
3. 打通百炼/千问调用
4. 打通数据库连接与元数据扫描
5. 实现安全 SQL 生成/校验
6. 实现即时问答流
7. 实现分析看板流
8. 接入网站 JSON 推送/落盘
9. 实现脱敏、日志、异常处理
10. 用真实问题联调验证

## 13. 验收标准
满足以下条件视为第一版可用：
- 群里 `@机器人` 能收到并处理消息
- 可回答“最近新增用户数”等直接问题
- 可回答“上周销售数据怎么样”类统计问题
- 可对“同比/原因分析”问题触发看板流
- 群里返回摘要并给出看板链接
- 结果默认脱敏且不显示 SQL
- 任意高危 SQL 请求都被拦截
- 服务重启后能恢复运行
