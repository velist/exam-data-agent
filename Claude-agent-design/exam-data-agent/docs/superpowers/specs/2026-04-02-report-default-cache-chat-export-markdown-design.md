# 设计文档：月报默认月份 + 缓存重建 + 对话导出按钮 + 详情 Markdown 排查

> 日期：2026-04-02 | 状态：已确认，待规划

---

## 背景

当前项目已经完成两项核心改造：
- 数据源已收敛为 `dws` 库
- 对话页图表已升级为与报表页一致的 ECharts 风格，并在代码中加入了 XLSX 下载能力

但线上/当前运行结果仍有四个待处理问题：
1. 点击月报时，月报默认月份仍然是当前自然月；当本月未过半时，应默认展示上月
2. 历史缓存需要清理并重新生成，确保缓存内容全部源自 `dws`
3. 用户当前在对话页没有看到导出按钮，需要定位到底是前端显示问题还是接口未返回可导出的表格数据
4. 报表详情中的 Markdown 格式没有正确体现，需要确认问题发生在流式内容、前端渲染，还是样式层

本轮工作的目标是以最小必要改动解决上述问题，不扩大到无关重构。

这四个问题不会进入同一个 implementation plan，而是拆成两个顺序执行的子项目：

### 子项目 A：月报默认月份 + 缓存重建

- 特点：规则明确、改动边界清楚、无需先做聊天链路诊断
- 产出：一份独立实施计划，完成后即可单独验证并收口

### 子项目 B：对话导出按钮 + 报表详情 Markdown
- 范围：对话页导出按钮复现与修复、报表详情 Markdown 复现与修复
- 特点：两项都依赖运行时观察与 Playwright 诊断，适合放在同一轮交互式排查中处理
- 产出：第二份独立实施计划，以运行期复现结果决定最小修复点

### 规划要求
- 后续 planning 必须按 **A → B** 顺序拆成两份计划，而不是用一份大计划覆盖全部内容
- 若在 planning 工具中必须一次输出，则必须明确分成两个相互独立的阶段，并分别定义完成条件与验证步骤

## 方案选择

### 方案 A（采用）
**前端修默认月 + 运营式清缓存重建 + 先诊断再修复导出/Markdown**

- 月报默认月份只在 `Report.tsx` 内收敛
- 缓存不改代码结构，按既有链路清理并重建
- 导出按钮和 Markdown 先用 Playwright 复现与定位，再做最小修复

**选择原因**
- 与本轮需求最贴合
- 改动面最小，回归风险最低
- 能直接区分“代码缺失”和“运行时数据/部署状态异常”

### 方案 B（不采用）
**把默认月份规则做成跨入口共享逻辑，并同步补充 URL 参数协议**

不采用原因：本轮只有一个月报入口，先在 `Report.tsx` 内收敛即可，避免无必要扩散。

### 方案 C（不采用）
**由后端提供推荐月份，并把缓存重建脚本化为专门命令**

不采用原因：超出当前最小修复范围，增加后端接口和运维复杂度。

## 现状确认

### 月报默认月份
- 月报入口位于 `frontend/src/pages/Chat.tsx`
- 进入月报时只导航到 `/report?type=monthly`
- 实际默认月份在 `frontend/src/pages/Report.tsx` 中初始化为 `dayjs().format("YYYY-MM")`

因此，在 2026-04-02 这样的日期进入月报时，当前会默认显示 2026-04，而不是预期的 2026-03。

### 缓存链路
当前缓存分为三层：
1. `backend/data/report_cache.json`
   - 由 `backend/services/report_cache.py` 在启动时读取/刷新
   - 当前预热查询均为 `dws` 表
2. `backend/data/query_cache/`
   - 由 `backend/services/query_cache.py` 按 SQL 哈希写入查询缓存
3. `frontend/public/cache/`
   - 由 `scripts/export_cache.py` 从 `query_cache` 复制并生成 `index.json`

因此，本轮不需要重写缓存逻辑，只需要清理旧缓存、重新生成，并验证内容中不再含有 `bigdata` 查询痕迹。

### 对话导出按钮
- 下载按钮代码已存在于 `frontend/src/components/ChatBubble.tsx`
- 按钮显示条件为：
  - `table.columns.length > 0`
  - `table.rows.length > 0`

这意味着“当前看不到按钮”并不一定是按钮组件缺失，也可能是当前回答没有返回非空表格数据。

### 详情 Markdown
- 报表详情页使用 `frontend/src/components/InsightText.tsx`
- 详情内容通过 `ReactMarkdown` 渲染
- 内容来自 `backend/services/insight.py` 的 SSE 流式输出

因此，问题可能在以下三处之一：
1. LLM 输出本身不是 Markdown
2. 前端渲染链路正常，但样式没有体现 Markdown 层级
3. 某些运行模式下详情流未正常返回

## 设计

## 一、月报默认月份规则

### 设计目标
当用户点击月报入口时：
- 每月 **1-14 日** 默认展示上月
- 每月 **15 日及以后** 默认展示当月

### 实现边界
- 只修改 `frontend/src/pages/Report.tsx`
- 不修改后端接口
- 不改变用户手动切换月份后的行为

### 设计细节
在 `Report.tsx` 中增加一个小型的默认月份计算函数，例如：
- 输入：当前日期
- 输出：默认月报月份字符串 `YYYY-MM`

逻辑：
- `date.date() < 15` 时，返回 `date.subtract(1, "month")`
- 否则返回当前月

页面初始化时，`month` 状态不再直接取 `dayjs().format("YYYY-MM")`，而是取该函数计算结果。

### 成功标准
- 在 2026-04-02 进入 `/report?type=monthly` 时，页面默认请求并展示 `2026-03`
- 在 2026-04-15 及以后进入时，默认请求并展示 `2026-04`

## 二、缓存清理与重建

### 设计目标
彻底清除历史缓存，重新生成一版仅基于 `dws` 的缓存结果，并确保前后端静态读取链路一致。

### 涉及缓存
- `backend/data/report_cache.json`
- `backend/data/query_cache/`
- `frontend/public/cache/`

### 执行顺序
1. 删除 `backend/data/report_cache.json`
2. 清空 `backend/data/query_cache/`
3. 清空 `frontend/public/cache/`
4. 启动后端，让 `report_cache.py` 重新从 `dws` 预热报表缓存
5. 触发需要保留的查询，重新写入 `query_cache`
6. 执行 `python scripts/export_cache.py`，重新导出静态缓存
7. 检查新缓存内容，确认不再含 `bigdata` 相关 SQL 或旧库痕迹

### 最小预热清单
为避免“缓存已重建但静态目录几乎为空”这种伪完成状态，至少需要重新生成以下两类缓存：

1. **报表缓存**
   - 周报：任选一个当前有效周日期，请求 `/api/report/weekly`
   - 月报：至少请求一个目标月份的 `/api/report/monthly`
   - 区间报表：至少请求一个合法区间的 `/api/report/range`

2. **查询缓存**
   - 至少执行一条销售类非时点查询，作为静态导出样本
   - 推荐基线问题：`3月各班次类型销量和销售额`
   - 该问题当前在 `backend/prompts/nl2sql.txt` 中已有稳定示例，预期会查询 `dws.dws_v_salesflow_dateil` 并返回非空表格结果

只有完成上述最小预热后，才允许执行静态缓存导出并认定缓存重建完成。

### 设计说明
- `report_cache.py` 当前预热表全部来自 `dws`，因此重建后的报表缓存天然满足目标
- `query_cache.py` 是运行期 SQL 结果缓存；只要旧缓存被清空，重建后就会只保留当前逻辑生成的 SQL 结果
- `export_cache.py` 只是把 `query_cache` 复制到前端静态目录，因此无需结构性修改

### 成功标准
- 三层缓存均已按顺序清理并重新生成
- 新的缓存文件中不出现 `bigdata.` SQL 痕迹
- 前端静态缓存目录与新的 `query_cache` 内容保持一致

## 三、对话导出按钮问题

### 设计目标
确保当聊天回答中包含“有表头且有数据行”的表格时，导出按钮稳定显示并可下载 XLSX。

### 设计策略
本问题先诊断再修复，不预设一定是前端按钮 bug。

### 诊断路径
使用 Playwright 实时复现一条固定的“应返回表格”的聊天请求，并检查：
1. 网络响应中 assistant 消息是否带 `table`
2. `table.columns` 是否非空
3. `table.rows` 是否非空
4. DOM 中是否渲染下载按钮
5. 按钮是否被样式、布局或条件渲染隐藏

### 基线输入
为避免诊断范围扩散，导出按钮问题固定使用以下基线问题复现：
- `3月各班次类型销量和销售额`

选择该问题的原因：
- 当前 `backend/prompts/nl2sql.txt` 中已有对应 SQL 示例
- 它属于明确的销售聚合查询，预期会命中 `dws.dws_v_salesflow_dateil`
- 按现有设计，应返回非空表格，因此能稳定区分“没表格”与“按钮没显示”两类问题

### 修复分支
#### 分支 A：接口返回了非空表格，但按钮未显示
修复 `ChatBubble.tsx` 的渲染条件或按钮样式/布局问题。

#### 分支 B：接口没有返回表格，或只返回空表
问题不在按钮本身，而在聊天回答结构或 SQL/结果转换链路。此时应把修复目标转到“确保该类问题产出表格数据”。

### 设计约束
- 不放宽为空表也显示下载按钮
- 不改变当前 XLSX 生成格式
- 保持“只有可导出数据时才显示按钮”的交互原则

### 成功标准
- 当 assistant 消息包含非空表格时，页面显示“下载 XLSX”按钮
- 点击按钮可成功下载，文件包含完整表头和数据行
- 空表或无表格回答时，按钮仍不显示

## 四、报表详情 Markdown 问题

### 设计目标
让报表详情中的 Markdown 层级被正确展示出来，至少覆盖：
- 标题
- 列表
- 加粗
- 段落换行

### 最小可验收示例
本轮验收至少要能稳定呈现以下结构：
- `## 数据概览`
- `## 关键发现`
- `## 产品建议`
- 列表项 `- **问题**：...`
- 加粗文本 `**具体数据和变化幅度**`

这些结构与当前 `backend/prompts/insight.txt` 中要求的输出格式一致，因此可以作为稳定验收基线。

### 设计策略
先用 Playwright 和接口观察确认问题落点，再做最小修复。

### 诊断路径
1. 打开报表详情页，观察 `InsightText` 的最终渲染 DOM
2. 检查 SSE 返回文本是否包含明确的 Markdown 标记（如 `#`、`##`、`-`、`**`）
3. 若返回文本有 Markdown 标记，再检查页面是否生成对应 HTML 结构
4. 若 HTML 结构已生成但视觉无层级，再检查 `report.css` 中 Markdown 样式

### 修复分支
#### 分支 A：流式文本本身不是 Markdown
调整 `backend/services/insight.py` 对应的提示词或生成要求，使洞察稳定输出 Markdown 结构。

#### 分支 B：返回文本是 Markdown，但前端展示为普通文本
修复前端渲染或容器处理问题，确保 `ReactMarkdown` 输出结构可见。

#### 分支 C：结构正确但样式弱或缺失
仅补 `report.css` 中的 Markdown 样式，不改流式链路。

### 设计约束
- 不把聊天页改为 Markdown 渲染
- 不重写 SSE 协议
- 优先修复“报表详情页的 Markdown 可读性”，不扩大为内容生成策略全面重做

### 成功标准
- 报表详情洞察中的标题、列表、加粗在页面上具有清晰层级
- 流式加载完成后，Markdown 内容可稳定复现
- 不影响现有状态提示卡片和流式游标逻辑

## 风险与控制

### 风险 1：默认月份逻辑只在前端生效
**控制方式：** 本轮只修入口体验，不扩展到后端统一口径；后续如新增其他月报入口，再统一抽取。

### 风险 2：缓存重建后需要重新生成查询缓存样本
**控制方式：** 在实施计划中明确“重建后要执行哪些代表性查询”，避免导出空静态缓存。

### 风险 3：导出按钮问题根因不在前端
**控制方式：** 先做 Playwright + 网络检查，再决定是否改 `ChatBubble.tsx`。

### 风险 4：Markdown 问题根因在 LLM 输出而非渲染
**控制方式：** 先看 SSE 原文，再决定改 prompt 还是改样式，避免盲改前端。
