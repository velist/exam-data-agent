# 设计文档：洞察分析优化

> 日期：2026-04-08
> 状态：已确认，待审阅
> 分支：`feat/insight-optimization`

---

## 背景

当前项目已经具备两条稳定的数据返回链路：

- 聊天页查数：优先命中预生成数据集与缓存，未命中时再走 NL2SQL + 查询
- 报表页洞察：先返回周报 / 月报 / 区间报表数据，再异步流式生成 AI 洞察

现状的主要问题不在“能不能查到数”，而在“查到数之后的分析是否足够懂业务”：

1. 聊天总结与报表洞察分属两套文案风格，输出结构不统一
2. 洞察层对医考业务节奏、考季阶段、考试类型语境利用不足
3. 现有对话意图建模主要服务于 NL2SQL，对“用户到底是想报数、看趋势、看异常还是要建议”建模不够
4. 建议与归因约束偏弱，容易出现“只有单值也输出泛化建议”的情况

结合当前项目方向，本次优化目标是：在不破坏现有查数速度与接口兼容性的前提下，让洞察输出更像懂考试培训业务的分析师，并继续补强对话意图建模。

---

## 目标

- 统一聊天洞察与报表洞察的分析骨架和业务判断方式
- 在洞察生成前注入考试类型 / 考季阶段 / 指标域 / 证据等级等上下文
- 把对话意图建模从“只服务 SQL 生成”升级为“同时服务查数与分析表达”
- 强化趋势、异常、建议的生成约束，避免无依据扩写
- 保持现有查数链路、缓存链路、前端流式协议兼容

---

## 非目标

- 不重写 `NL2SQL` 主流程
- 不重构 `dataset_router` / `report_cache` / `report.py` 的主体逻辑
- 不新增数据库表、远程知识库或复杂规则引擎
- 不修改聊天与报表接口的返回结构
- 不在本次改动中直接部署上线；仅在分支完成实现与本地验证

---

## 方案对比

### 方案 A（采用）
只增强洞察层与意图摘要层，不改查数主链路。

**做法：**
- 保持 `chat_stream -> dataset_router/NL2SQL -> query -> table` 主链路不变
- 在聊天总结与报表洞察进入 LLM 前统一构造分析上下文
- 将意图建模从“是否追问、是否继承口径”扩展到“用户想看数据 / 趋势 / 异常 / 建议什么”
- 用轻量证据分级与异常语义分层约束输出

**优点：**
- 风险最低
- 不影响缓存与秒出表格体验
- 能同时改善聊天页与报表页分析质量
- 本地测试范围清晰

**缺点：**
- 第一版更偏“分析质量升级”，不引入复杂指标派生与规则引擎

### 方案 B（不采用）
在方案 A 基础上增加较重的异常规则引擎和模板系统。

**不采用原因：**
第一版体量过大，容易把本次目标从“提升分析质量”扩展成“重做分析系统”。

### 方案 C（不采用）
把考试类型、考季、行业动态做成独立知识库，并深入参与路由和查询阶段。

**不采用原因：**
超出本次分支最小可行范围，且会增加接口与维护复杂度。

## 设计

### 一、架构边界

本次优化只落在 **洞察生成层 + 意图摘要层**，不改现有查数主链路。

#### 保持不动的部分

- `backend/services/chat_stream.py` 中的快速路由 / NL2SQL / 查询 / 回表编排
- `backend/services/dataset_router.py` 中的数据集快速路由主体逻辑
- `backend/services/report.py` 中的周报 / 月报 / 区间报表聚合结构
- 前端消费协议：聊天页继续消费 `table + answer_chunk`，报表页继续消费 `/api/insight/stream`

#### 本次主要改动的部分

- 聊天洞察：`backend/services/chat.py`
- 报表洞察：`backend/services/insight.py`、`backend/prompts/insight.txt`
- 新增一层轻量上下文构造模块，用于统一组织意图、业务场景与生成约束

### 二、统一数据流

#### 聊天链路

现有聊天链路已经具备：
- 追问识别
- 上轮口径继承
- 路由命中优先

本次新增一份 **洞察意图摘要**，在总结阶段与现有追问建模结果一起使用：

1. 用户消息进入后，先沿用当前追问/新问题判断
2. 若未命中快速路由，则继续走现有 NL2SQL 与查询
3. 查询返回表格后，不直接把表格裸喂给总结 prompt
4. 先构造一份分析上下文，再与表格数据一起传给总结 LLM

这份分析上下文至少包含：
- 查询意图：新问题 / 追问 / 维度拆分 / 条件收缩 / 趋势对比 / 明细排名
- 分析意图：用户更偏向报数、看趋势、看异常、要建议中的哪类
- 上下文继承信息：是否沿用上一轮统计对象、时间范围、筛选与分组
- 指标域：用户 / 留存 / 行为 / 付费 / 销售 / 客服
- 当前考季阶段
- 证据等级

#### 报表链路

报表链路没有自然语言追问，但同样要复用统一分析框架：

1. `report.py` 生成结构化报表数据
2. `insight.py` 先根据报表类型、指标分布、变化字段构造分析上下文
3. 再把分析上下文 + 报表数据送给统一风格的洞察 prompt

这样聊天洞察与报表洞察的区别只剩“入口不同”，而不是“分析人格不同”。

### 三、意图建模增强

本次不重做整套 NL2SQL，只增强与分析强相关的部分。

#### 现有可复用能力

当前 `chat.py` 已能识别：
- 是否为追问
- 是否是分析方式补充（环比 / 同比 / 趋势）
- 是否是结果范围收缩（如退款类）
- 是否是分组维度变化（如按周 / 按系列）

#### 本次补强方向

在现有基础上，新增更高一层的意图标签，用于同时服务“查数”和“分析”：

- **查询意图标签**
  - new_query
  - follow_up_analysis
  - follow_up_filter
  - follow_up_grouping
  - detail_or_ranking
- **分析意图标签**
  - value_only
  - trend_focus
  - anomaly_focus
  - action_focus
- **业务语境标签**
  - user_growth
  - retention
  - learning_behavior
  - payment
  - sales
  - customer_service

#### 设计原则

- 不引入复杂的多轮 agent 状态机
- 优先复用现有 message + history + table_data 即可推断的信息
- 意图标签允许采用可组合标签，而不是强制单标签；实现时以“一个主标签 + 少量补充标签”为主，避免过度复杂
- 若无法确定分析意图，默认保守：先报核心数据，再少做归因
- 若无法识别考试类型或考季阶段，则直接跳过阶段判断注入，只按数据证据生成洞察，不做额外猜测

### 四、统一输出骨架

聊天总结与报表洞察统一遵守以下骨架：

1. **核心结论**
   - 直接回答当前最关键的数据结果
2. **趋势 / 异常**
   - 仅当存在同比、环比、多期趋势或显著对比依据时输出
3. **业务判断**
   - 结合考试阶段、业务模块、用户链路做解释
4. **建议动作**
   - 仅在证据充分时给，且必须落到具体模块或动作

#### 骨架约束

- 聊天页可以更短，但结构不变
- 报表页可以更完整，但判断逻辑不另起炉灶
- 无证据时，允许停在“核心结论”或“核心结论 + 观察”，不强行输出建议

### 五、考季阶段与业务上下文注入

本次会把已调研的医考周期信息转成轻量的阶段标签，作为分析背景输入，而不是输出给用户看的硬编码规则。

阶段标签按“考试类型 × 时间阶段”组织，但第一版保持轻量，优先支持这些语义：

- 报名启动期
- 缴费确认期
- 系统备考期
- 冲刺提分期
- 查分转化期
- 淡季蓄水期

使用原则：
- 作为业务判断与建议的背景信息
- 不能覆盖实际数据本身
- 不能在没有数据支撑时，仅凭考季阶段下结论

### 六、证据等级与异常语义分层

本次增加轻量证据分级，仅作为生成约束，不修改接口结构。

#### 证据等级

- **高证据**
  - 存在明确环比 / 同比 / 多期趋势 / 多维拆分
  - 示例：表格中直接含有环比列、同比列，或同一指标有最近多周走势
  - 允许输出较强的趋势判断和动作建议
- **中证据**
  - 有单期值 + 结构拆分
  - 示例：单月按渠道、按班次、按系列拆分，能看结构但缺少明确时间对比
  - 可输出结构观察和重点关注点，但少做因果判断
- **低证据**
  - 只有单指标值或单条结果
  - 示例：只查到“上月销售总额”一个数字，或只返回单行单列结果
  - 只报数或轻量观察，不扩写趋势和建议

#### 异常语义分层

- 有明确变化依据且幅度较大：使用“重点关注”
- 有变化但不足以定义为明显异常：使用“有变化 / 需持续观察”
- 无变化依据：不写异常

这部分先做轻量语义约束，不引入复杂阈值引擎。

### 七、建议动作约束

建议必须同时满足以下条件：

1. 与当前指标域匹配
2. 可落到考试宝典已有业务模块
3. 有当前数据支撑

建议映射方向示例：
- 付费问题 → 首购承接、班次页转化、客服转化链路
- 学习行为问题 → 刷题、模考、课程、AI答疑入口与承接
- 留存问题 → 新用户首日引导、连续学习任务、内容触达
- 销售问题 → 渠道结构、班次结构、客服/合作商转化差异

若证据不足，则不输出建议动作。

## 验证方式

### 自动化测试

后端重点验证：

1. 追问意图仍能继承上轮口径
2. 查询链路与 SSE 事件结构保持兼容
3. 只有单值时，总结不会强写趋势或建议
4. 有明确对比信息时，总结可输出趋势 / 异常判断
5. 报表洞察与聊天洞察都走统一的分析约束

前端只做回归确认，保证现有接口消费不受影响。

### 手工本地验证

重点覆盖：

1. 聊天页普通查数
2. 聊天页追问“环比 / 同比 / 按维度拆分”
3. 周报 / 月报 / 区间报表洞察输出
4. 图表 / 表格先展示，AI 洞察继续异步生成
5. 仅在分支上完成实现与测试，不提前上线

## 风险与控制

### 风险 1：把分析逻辑做重，反而影响当前稳定链路
**控制方式：** 仅在洞察层和上下文层改动，查数主链路不动。

### 风险 2：规则过多，导致实现复杂度膨胀
**控制方式：** 第一版只做轻量标签与证据分级，不做复杂规则引擎。

### 风险 3：聊天与报表洞察仍出现两套风格
**控制方式：** 两条链路共享同一套分析骨架与上下文构造方法。

### 风险 4：建议动作脱离当前业务模块
**控制方式：** 明确限制建议只能落到现有模块：刷题、模考、课程、AI答疑、班次页、客服/渠道链路。

### 风险 5：考季阶段被过度使用，压过真实数据
**控制方式：** 考季标签只作为解释背景，不能替代数据证据。

---

## 验收标准

完成后应满足：

1. 聊天页与报表页的洞察输出结构明显统一
2. 洞察能体现考试培训业务语境，而不是通用 BI 套话
3. 对话意图建模比当前更完整，既服务查数，也服务分析表达
4. 无同比/环比/趋势证据时，不编造趋势或泛化建议
5. 现有查数链路、缓存命中、前端异步展示体验保持不变
6. 所有实现仅在 `feat/insight-optimization` 分支完成，并先通过本地测试
