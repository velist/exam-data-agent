## 1. 项目一句话说明

这是一个面向“考试宝典”业务的数据分析助手项目，核心能力是：

1. 用自然语言进行业务数据查询
2. 在聊天页输出结构化表格、图表和文字结论
3. 生成周报 / 月报 / 自定义区间报告
4. 对报告进一步生成 AI 洞察
5. 同时支持“完整后端模式”和“静态缓存模式”两种运行方式

可以把它理解为：

- 前端：一个面向业务人员的数据问答与报表界面
- 后端：一个把自然语言转成 SQL、执行查询、组织报告、输出 AI 洞察的服务层
- 缓存层：把高频问题和报表结果固化下来，既减轻数据库压力，也支持静态部署

## 4. 项目业务边界与核心约束

这部分是最容易被新同学忽略、但实际最关键的内容。

### 4.1 数据源约束：现在是 dws-only

当前项目的数据源约束已经明确收敛为：

- **只读 dws 库**
- 不应再回退到其他库
- 报表预热 SQL、聊天查询白名单、销售分析基线都要按这个约束继续维护

需要重点知道的销售数据表：

- `dws.dws_v_salesflow_dateil`

### 4.2 销售口径约束

当前销售口径不是随意聚合，已经明确：

- **销量 = `COUNT(*)`**
- **销售额 = `SUM(销售金额)`**

后续凡是涉及“销量”“销售额”的 SQL、报表、洞察解释，都必须遵守这个口径。

### 4.3 业务周定义不是自然周

当前业务周定义为：

- **周六 ~ 周五**

所以凡是周报、环比周数据、按周趋势，不要按默认周一~周日理解。

### 4.4 静态模式与完整模式的差异

前端存在静态模式判断逻辑：

- 没有后端时，会进入静态模式
- `sendChat()` 和 `streamChat()` 在静态模式下都会直接返回 `STATIC_MODE` 错误提示
- `streamInsight()` 在静态模式下也不会真正请求后端
- 因此当前静态模式**不是完整实时对话替代品**，而更接近“只读展示 / 预生成内容消费”模式

所以“某个页面看起来功能不全”时，第一件事要先判断当前到底是不是完整后端模式。

## 5. 系统架构总览

### 5.1 逻辑架构

```text
用户
  ↓
React 前端
  ├─ 聊天页（问题输入、流式消息、表格、图表、XLSX）
  └─ 报表页（周报 / 月报 / 区间报表 + AI 洞察）

  ├─ /api/chat
  ├─ /api/chat/stream
  ├─ /api/report/weekly
  ├─ /api/report/monthly
  ├─ /api/report/range
  └─ /api/insight/stream
  ↓

  ├─ NL2SQL / query execution
  ├─ 报表聚合
  ├─ 洞察生成
  ├─ 查询缓存
  └─ 报表缓存
  ↓
MySQL / AnalyticDB（dws）
```

### 5.2 前后端关系

当前不是完全分离部署思维，而是：

- `frontend/dist` 会被后端静态挂载
- `backend/main.py` 同时提供 API 与前端静态文件 fallback

所以本地完整模式跑起来以后，用户访问的是统一入口。

- “上周注册用户情况”
- “付费转化率趋势”

系统会返回：

- 表格
- 可能的图表
- 总结性文本
- 某些场景下可导出 XLSX

#### 代码理解顺序

1. `frontend/src/pages/Chat.tsx`
2. `frontend/src/api.ts`
3. `backend/main.py`
4. `backend/services/chat_stream.py`
5. `backend/services/chat.py`
6. `backend/prompts/nl2sql.txt`

### 7.2 对话流式渲染链路

这里有一个很重要的历史设计点：

- 前端不是一次性等全量回答返回
- 而是按 SSE 事件流逐步更新状态
- 表格和文本可能不是同一时刻到达

所以如果未来又遇到“表格丢了”“导出按钮没了”“done 后内容消失”，优先检查：

- `chat_stream.py` 是否发送了 `table`
- `chatMessageUtils.ts` 是否把 `table` 保住了
- 是否被后续事件覆盖

### 7.3 报表链路

报表有三类：

- 周报
- 月报
- 自定义区间报表

关键点：

- 周报：基于业务周（周六~周五）
- 月报：现在有“月中前默认上月”的规则
- 区间报表：用于复盘任意一段时间

代码理解顺序：

1. `frontend/src/pages/Report.tsx`
2. `backend/main.py`
3. `backend/services/report.py`
4. `backend/services/report_cache.py`

### 7.4 洞察链路

洞察不是前端自己总结，而是后端把报告数据组织后，再调用模型流式生成。

代码理解顺序：

1. `frontend/src/components/InsightText.tsx`
2. `frontend/src/api.ts` 中 `streamInsight`
3. `backend/services/insight.py`

1. **DOM 结构对不对**（heading / list / strong 是否生成）
2. **视觉层级明显不明显**（CSS 问题）

不要一上来就改 prompt，也不要一上来就改前端组件。

## 8. 缓存设计：两层缓存架构（2026-04-03 已落地）

这个项目不是”每次都实时查库”，而是采用**两层缓存 + 意图路由**架构：

### 8.1 架构总览

```text
用户提问
  │
  ├─ dataset_router 意图路由器（关键词匹配，<1ms）
  │   │
  │   ├─ 命中高频主题 ──→ dataset_cache / report_cache（内存读取）──→ 表格秒出 + 异步 AI 总结
  │   │
  │   └─ 未命中 ──→ NL2SQL ──→ query_cache 检查 ──→ 命中则返回 / 未命中则实时查 DB
  │
  └─ 报表页 ──→ report_cache（已有机制，不变）
```

### 8.2 第一层：预生成数据集 + 报表内存缓存（高频秒出）

#### 8.2.1 dataset_cache — 销售 / 客服预聚合

位置：
- `backend/services/dataset_cache.py`
- 磁盘文件：`backend/data/dataset_cache.json`

包含 6 个预聚合数据集：
- `sales_monthly_total` — 月度销量 / 销售额汇总
- `sales_monthly_by_class` — 按月 + 班次类型
- `sales_monthly_by_channel` — 按月 + 渠道类型
- `sales_monthly_by_series` — 按月 + 系列名称
- `cs_by_theme` — 客服按问题主题（最近 90 天）
- `cs_by_series` — 客服按系列

机制：启动时磁盘秒开 → 后台线程从 DB 刷新 → 每 6 小时自动刷新

#### 8.2.2 report_cache — 5 张 DWS 周报表

位置：
- `backend/services/report_cache.py`
- 磁盘文件：`backend/data/report_cache.json`

已有机制不变，同时也被 dataset_router 用于聊天查数（活跃 / 付费 / 留存 / 行为类问题）。

#### 8.2.3 dataset_router — 意图路由器

位置：`backend/services/dataset_router.py`

在 `chat_stream.py` 的 NL2SQL 之前拦截：
- 基于关键词匹配判断意图（销售 / 活跃 / 付费 / 留存 / 客服 / 概览）
- 正则提取时间（”3月” / “上月” / “最近N周”）
- 追问场景（环比 / 同比 / 指代词）不走快速路由，回退 NL2SQL

### 8.3 第二层：NL2SQL + 查询缓存（长尾兜底）

位置：
- `backend/services/query_cache.py`
- 目录：`backend/data/query_cache/`

对 dataset_router 未命中的问题，仍走原链路：NL2SQL → SQL 校验 → 执行查库 → 结果写入 query_cache。

### 8.4 前端静态缓存（备用）

位置：
- 导出脚本：`scripts/export_cache.py`（已升级：自动过滤 bigdata + 导出预聚合数据集）

导出脚本现在会：
1. 过滤掉所有 `bigdata` SQL 缓存
2. 同时导出 `dataset_cache.json` 中的预聚合数据集

在完整后端模式下，静态缓存为备用方案。

### 8.5 缓存清理状态（2026-04-03）

- `bigdata` 脏缓存已彻底清理（query_cache + static cache + index.json）
- 当前 `backend/data/query_cache/` 中 6 个文件全部是 dws 查询
- `frontend/public/cache/` 已重新导出，只含 dws 数据

### 8.6 缓存接手时的注意事项

5. `scripts/export_cache.py` 已自动过滤 bigdata SQL，不需要手动检查

## 9. 当前项目最重要的历史难点

下面这些不是“代码里看一眼就懂”的信息，而是接手者最应该知道的难点。

### 9.1 自然语言查询不是单一 SQL 问题，而是“口径约束 + 安全 + 追问继承”问题

难点不在“生成 SQL”本身，而在：

- 用户提问很模糊
- 追问经常不完整
- 不同轮对话需要继承上轮范围
- 同时要防止生成危险 SQL

这决定了你不能把这个项目简单理解成“LLM + SQL”。

### 9.2 页面现象不一定等于前端问题

本轮就有两个典型例子：

#### 例 1：导出按钮“看不见”

最后定位结果不是前端坏了，而是：

- 对固定基线问题，它本来就正常
- 真正要看的是接口是否返回了非空 `table`

#### 例 2：Markdown “没加载好”

最后定位结果也不是组件没渲染，而是：

- 真实页面会生成正确的 heading/list/strong 结构
- 要警惕的是不同环境、不同流式结果、不同观察时机

### 9.3 缓存与实时能力并存，容易把问题看串

这个项目一大难点是：

- 同一个页面，有时走实时 API
- 有时走静态缓存
- 有时报表是缓存出来的，但洞察是实时流式生成的

如果未来有人说“这个页面不对”，先分清它走的是哪条链路。

### 9.4 文档与代码可能存在轻微不一致

当前已确认的一个典型点：

- 代码里的 `backend/config.py` 实际读取的是 **仓库根目录 `.env`**
- 但历史 README 曾写成 `backend/.env`

接手时请优先相信当前代码，再回头修正文档。

## 15. 最后一句提醒

这个项目最容易误判的地方不是“代码复杂”，而是：

- 现象像前端，其实是数据链路
- 现象像样式，其实是环境模式
- 现象像组件 bug，其实是某次返回值为空

所以接手时，先分清：

1. 当前运行模式是什么
2. 当前数据是不是缓存来的
3. 当前接口有没有真正返回结构化数据
4. 当前问题是否能用固定基线稳定复现

做到这四步，接手效率会非常高。
